Introduction - Big O Notation

Big O Notation

Big O notation is a way to formalize fuzzy counting. This means it allows us to talk formally about how the runtime of an algorithm grows as the inputs grow. With big O notation, we tend to look at the bigger picture of measuring the quality of our code by measuring the number of operations the server has to perform to run our code. We could say that an algorithm is O(f(n)) if the number of simple operations a server has to do is eventually less than a constant times f(n), as n increases where f() denotes the function and n denotes the input of function f(). This brings about 4 categorizations of Big O where:

  1. f(n) could be linear O(f(n) = n) this means runtime is proportional to the value of n
  2. f(n) could be quadratic O(f(n) = n^2) this means we have two linear functions where n is run twice as much(loop within a loop)
  3. f(n) could be constant O(f(n) = 1) this means runtime is always roughly constant
  4. f(n) could be something entirely different which means it doesn't fall in any of the above notations
Consider the examples below where we calculate the sum of all numbers from 1 up to (and including) some number n.

Example One

Here we have way more number of operations because of the loop. This means the number of operations grow proportional to the value of n thus this runs slower. This means this has a big O of n denoted as O(n).

  1. const addUpToFirst = (n) => {
  2.     let total = 0;
  3.  
  4.     for (let index = 1; index <= n; index++) {
  5.         total += index;
  6.     }
  7.  
  8.     return total;
  9. };
Example Two

Here we have 3 operations(1 multiplication, 1 addition and 1 division) that the server performs regardless of the size of n. This means this would ran faster than example one above. This has a big O of 1 denoted by O(1).

  1. const addUpToSecond = (n) => {
  2.     return n * (n + 1) / 2;
  3. };

To see this in action open the console to see how long it takes for both examples to run with an input of 1,000,000,000

Simplifying Big O

With the above examples we come to see that there are rules to simplifying expressions for Big O notations where:

  1. Arithmetic operations are always constant
  2. Variable assignment is also constant
  3. Accessing an element in an array(by index) or object(by key) is constant
  4. In a loop, the complexity is the length of the loop times the complexity of whatever happends inside the loop
Space Complexity(auxilliary space complexity)

In the above sections, the main focus has been on time complexity where we analyse the runtime of an algorithm as the size of the inputs increase. However, Big O notation can also be used to analyse space complexity which checks on how much additional memory do we need to allocate in order to run an algorithm. There are rules guiding space complexity which include:

  1. Most primitives such as booleans, numbers, undefined and null are constant space
  2. Strings require O(n) space where n is the string length. This means bigger the number of characters there are in a string the more space it takes
  3. Reference types are generally O(n) space, where n is the length(for arrays) or the number of keys(for objects)
